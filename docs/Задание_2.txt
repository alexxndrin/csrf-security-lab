ЗАДАНИЕ №2: ПРОЕКТИРОВАНИЕ АРХИТЕКТУРЫ CSRF-ЗАЩИТЫ ДЛЯ ПРИЛОЖЕНИЯ «ДОСКА ОБЪЯВЛЕНИЙ»
1. Место хранения токена на сервере
     В данной архитектуре CSRF-токен должен храниться в сессии пользователя (Session Storage)
     Необходимость сессии, а не базы данных заключается в том, что:
• CSRF-токен проверяется при каждом небезопасном запросе (POST, PUT, DELETE). Обращение к оперативной памяти, где обычно хранятся сессии (например, через Redis или Memcached), происходит значительно быстрее, чем запрос к дисковой БД. Это снижает нагрузку на систему при высокой посещаемости доски объявлений.
• Жизненный цикл CSRF-токена неразрывно связан с текущим сеансом работы пользователя. Сессия автоматически инвалидируется при выходе пользователя или по таймауту. Если хранить токены в БД, потребуется создание дополнительных механизмов очистки устаревших записей, чтобы избежать «раздувания» базы.
• Хранение в сессии гарантирует, что токен привязан к конкретному идентификатору сессии (Session ID). Это исключает возможность использования токена одного пользователя для подтверждения действий другого.
2. Жизненный цикл CSRF-токена
     Схема процесса:
     1. Генерация: при GET-запросе к странице с формой (например, /ads/create) сервер создаёт уникальную криптографически стойкую строку с помощью функции random_bytes() и сохраняет её в сессии пользователя: $_SESSION['csrf_token'] = $token.
     
     2. Передача клиенту: сервер вставляет сгенерированный токен в HTML-код страницы внутри формы в виде скрытого поля: <input type="hidden" name="_csrf_token" value="сгенерированный_токен">.
     3. Хранение в форме:токен находится в DOM-дереве страницы внутри скрытого поля. Браузер хранит его как часть HTML-формы до момента отправки.
     4. Отправка с запросом: при заполнении и отправке формы браузер формирует POST-запрос и автоматически включает в его тело значение скрытого поля с токеном.
     5. Проверка: сервер получает POST-запрос, извлекает токен из $_POST['_csrf_token'] и сравнивает его с токеном из $_SESSION['csrf_token'] с помощью функции hash_equals() (защита от атак по времени).
     6. Что делать при несовпадении:
     Если токены совпадают: запрос считается легитимным, сервер выполняет действие (сохраняет данные в БД) и возвращает HTTP 302 (редирект) или HTTP 200 (успех).
     Если токены не совпадают (или токен отсутствует): сервер отклоняет запрос, не выполняет никаких действий с данными и возвращает HTTP 403 Forbidden (доступ запрещён). В ответе может быть выведено сообщение об ошибке безопасности.
3. Псевдокод базового класса CsrfGuard
      class CsrfGuard {
      
          public static function generate() {
              $token = bin2hex(random_bytes(32));
              $_SESSION['csrf_token'] = $token;
              return $token;
          }
          
          
          public static function embed() {
              $token = self::generate();
              return '<input type="hidden" name="_csrf_token" value="' . $token . '">';
          }
          
          
          public static function validate($inputToken) {
              if (!isset($_SESSION['csrf_token'])) {
                  return false;
              }
              
              if (empty($inputToken)) {
                  return false;
              }
              
              return hash_equals($_SESSION['csrf_token'], $inputToken);
          }
      }
4. HTTP-ответы:
     При успешной проверке:
     HTTP 200 OK (если остаёмся на той же странице с сообщением об успехе) или HTTP 303 See Other (для перенаправления на другую страницу, например, на список объявлений). Сервер подтверждает, что запрос корректен, личность отправителя верифицирована через владение секретным токеном, и действие успешно выполнено.
     При неудачной проверке:
     403 Forbidden (Доступ запрещен). Ошибка проверки CSRF-токена означает, что запрос не является доверенным. Статус 403 наиболее точно указывает на то, что сервер понял запрос, но отказывается его выполнять из-за нарушения прав доступа или политик безопасности. Использование 401 Unauthorized здесь некорректно, так как пользователь может быть авторизован (иметь активный логин), но сама «посылка» данных скомпрометирована.
