ЗАДАНИЕ №1: АНАЛИЗ БЕЗОПАСНОСТИ MVC-ПРИЛОЖЕНИЯ «ДОСКА ОБЪЯВЛЕНИЙ»
     Даны маршруты приложения:
• GET /ads — список объявлений
• POST /ads/create — создать объявление (требует авторизации)
• POST /ads/delete — удалить объявление (требует авторизации)
• POST /profile/update — обновить профиль пользователя
     Необходимо проанализировать маршруты на уязвимость к CSRF-атаке и описать сценарий атаки для двух наиболее опасных маршрутов.
1. Анализ маршрутов на уязвимость к CSRF-атаке
     CSRF (межсайтовая подделка запроса) — это тип атаки, при которой сторонний ресурс заставляет браузер пользователя отправить запрос на целевой сервер от его имени.
     Маршрут: GET /ads (список объявлений)
     Уровень риска: не уязвим.
     Метод GET предназначен только для получения данных. Согласно стандартам безопасности, операции чтения не должны менять состояние системы, поэтому «подделка» этого запроса не принесет вреда данным пользователя.
     Маршрут: POST /ads/create (создать объявление)
     Уровень риска: уязвим.
     Запрос использует метод POST и изменяет состояние базы данных (создает новую запись). Если приложение не проверяет наличие уникального CSRF-токена, злоумышленник может создать скрытую форму на своем сайте, которая отправит запрос на публикацию объявления от лица авторизованного пользователя.
     Маршрут: POST /ads/delete (удалить объявление)
     Уровень риска: уязвим.
     Это критическая операция. Отсутствие защиты позволяет атакующему инициировать удаление контента пользователя без его ведома через сторонний скрипт.
     Маршрут: POST /profile/update (обновить профиль)
     Уровень риска: уязвим.
     Маршрут позволяет изменять персональные данные. Без CSRF-защиты злоумышленник может принудительно сменить настройки профиля пользователя (например, привязать другой номер телефона или email).
2. Сценарии атаки для наиболее опасных маршрутов
     Сценарий 1: удаление объявления (/ads/delete)
     Злоумышленник создает вредоносную страницу или рекламный баннер, в который встроен скрытый HTML-код с формой. Форма настроена на автоматическую отправку (autosubmit) при загрузке страницы.
     Если пользователь авторизован в приложении «Доска объявлений» в том же браузере, то, после захода на сайт злоумышленника, браузер видит запрос к POST /ads/delete и автоматически прикрепляет к нему активные файлы куки (cookies) пользователя.
     В итоге сервер принимает запрос как подлинный и удаляет объявление. Пользователь теряет свои данные, не понимая причины.
     Сценарий 2: обновление профиля (/profile/update)
     Злоумышленник с помощью социальной инженерии заманивает пользователя на специально подготовленный сайт. На сайте срабатывает скрипт, который отправляет скрытый POST-запрос на /profile/update с заранее заданными параметрами (например, новый почтовый адрес, контролируемый хакером).
     В итоге почта в профиле пользователя меняется. Злоумышленник получает возможность запросить сброс пароля на свой email и полностью захватить аккаунт.
